In top level of the model, there are three layers:
1) ICD11 Ontological Layer (analogous to Dictionary with role of definition)
2) ICD11 Category Layer (analogous to categorization and knowledge base) and
3) ICD11 Linearization Layer (analogous to different serialization views for publications).

In each layer, there is a corresponding core entity. We use "ConceptDescriptor" in ICD11 Ontological Layer, "ICD11_Category" in ICD11 Category Layer, "ICD11 Linerization Entity" in ICD11 Linearization Layer. The entity in each layer shares a common superclass "ICD11_Entity".

The relationships among the entities in three layers are described as:

Every ICD11_Category has exactly one definitions of type ConceptDescriptor.
Every ConceptDescriptor has exactly one defined category of type ICD11_Category.

Every ICD11_Category has one or more views of type ICD11_Linearization_Entry.
Every ICD11_Linearization_Entity has exactly one view_source of type ICD11_Category.

Every ICD11_Linearization_Entity has exactly one definition of type ConceptDescriptor.
Every ConceptDescriptor has one or more defined entities of type ICD11_Linearization_Entity.

Note by Alan Rector: RELATIONSHIP BETWEEN LAYERS - "PROJECTION"
The layers have different structure and different invariants.  Therefore,  the relationship between layers will lossy. Therefore the relationship between them should be an explicit "projection" or "view mechanism", i.e. a function that carries the appropriate information from one layer to the next, possibly requiring additional inputs.  Ideally this should be expressed in a declarative way, but for now I we can keep it simple.  One of the functions of "projection" should be to propagate (pointers to) Presentation Objects between layers where appropriate.
